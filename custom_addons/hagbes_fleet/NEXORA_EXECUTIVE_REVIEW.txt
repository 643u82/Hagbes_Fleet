┌─────────────────────────────────────────────────────────────────────────────┐
│ ISSUE CRIT-005: PROPERTY MANAGER APPROVAL NOT RESTRICTED                   │
├─────────────────────────────────────────────────────────────────────────────┤
│ Status: CONFIRMED - ACTIVE IN CURRENT CODE                                  │
│ Verification Confidence: HIGH (Direct XML Evidence)                         │
└─────────────────────────────────────────────────────────────────────────────┘

ROOT CAUSE:
Approval flows use role-based authorization (res.groups) without organizational
context. Any user in group_property_manager can approve ANY request globally,
regardless of department, branch, or organizational unit.

EVIDENCE LOCATION:
- File: data/fleet_approval_flows.xml
- Lines: 19-23 (Fleet Assignment), 67-71 (Maintenance), 115-119 (Disposal)
- Code: <field name="role_id" ref="hagbes_fleet.group_property_manager"/>

TECHNICAL DETAILS:
Three approval workflows affected:
1. Fleet Vehicle Assignment approval
2. Fleet Maintenance approval  
3. Fleet Vehicle Disposal approval

Each uses group_property_manager as approver role without domain filtering.

SECURITY RISK LEVEL: ★★★★★ CRITICAL (5/5)
├─ Authorization Bypass: Any Property Manager approves any request
├─ No Separation of Duties: Cross-department approval possible
├─ Compliance Violation: SOX, ISO 27001 requirements not met
├─ Fraud Risk: Collusion between requester and any Property Manager
└─ Audit Trail Weakness: Cannot determine if approver was authorized

BUSINESS IMPACT: ★★★★☆ HIGH (4/5)
├─ Regulatory Compliance Risk
├─ Internal Control Failure
├─ Asset Misappropriation Risk
├─ Organizational Boundary Violation
└─ Reputation Risk (if exploited)

PRODUCTION READINESS IMPACT: ★★★★★ CRITICAL (5/5)
├─ BLOCKS PRODUCTION DEPLOYMENT
├─ Must be fixed before go-live
├─ Creates audit liability
└─ Violates basic security principles

RECOMMENDED SOLUTION:
Option 1 (Quick Fix - 2 days):
Extend approval.step model to add approver_domain field that filters
eligible approvers by department/organizational unit.

Option 2 (Comprehensive - 1 week):
Create department-specific security groups (e.g., group_property_manager_dept_a)
and assign users to appropriate groups based on their organizational scope.

Option 3 (Enterprise - 2 weeks):
Implement context-aware approval authorization with custom validation logic
that checks approver's organizational context against record context.

ESTIMATED IMPLEMENTATION EFFORT:
├─ Quick Fix (Option 1): 2-3 developer days
├─ Comprehensive (Option 2): 5-7 developer days
└─ Enterprise (Option 3): 10-15 developer days

RECOMMENDED APPROACH: Option 2 (Comprehensive)
Provides long-term maintainability and clear authorization model.

TESTING STRATEGY:
1. Unit Tests: Verify approvers filtered by department
2. Integration Tests: Cross-department approval blocked
3. Security Tests: Unauthorized approval attempts fail
4. Regression Tests: Authorized approvals still work
5. Compliance Tests: Audit trail shows correct authorization context

┌─────────────────────────────────────────────────────────────────────────────┐
│ ISSUE CRIT-001: REQUESTER ACL TOO PERMISSIVE                               │
├─────────────────────────────────────────────────────────────────────────────┤
│ Status: PARTIALLY CONFIRMED - SECURITY CONCERN                              │
│ Verification Confidence: MEDIUM (Functional but architecturally weak)      │
└─────────────────────────────────────────────────────────────────────────────┘

ROOT CAUSE:
Fleet.requisition ACL grants write permission to ALL requesters globally.
Submission works via sudo() bypass, but direct write operations are possible.

EVIDENCE LOCATION:
- File: security/ir.model.access.csv, Line 24
- Code: perm_write=1 for group_fleet_requester
- File: models/fleet_requisition.py, Line 714
- Code: rec.sudo()._trigger_approval()

TECHNICAL DETAILS:
The ACL is broader than necessary:
- Requesters can write to ANY requisition record (subject to record rules)
- sudo() in submission action bypasses all security
- Write permission not needed if sudo() is used correctly
- Creates potential for security misconfiguration

SECURITY RISK LEVEL: ★★★☆☆ MEDIUM (3/5)
├─ ACL Overpermissive: Write granted but not needed
├─ Record Rules: Provide protection (defense in depth)
├─ Sudo Usage: Bypasses checks (correct but risky pattern)
└─ Direct Manipulation: Possible via Python shell/API

BUSINESS IMPACT: ★★☆☆☆ LOW (2/5)
├─ Functional: Submission works correctly
├─ Risk: Only exploitable by technical users
├─ Record Rules: Provide practical protection
└─ Limited Exposure: Requires system access

PRODUCTION READINESS IMPACT: ★★☆☆☆ LOW (2/5)
├─ Does NOT block deployment
├─ Can be postponed to Phase 2
├─ Best practice violation, not critical bug
└─ Should be fixed for security hardening

RECOMMENDED SOLUTION:
Remove write permission from requester ACL:

Change security/ir.model.access.csv Line 24:
FROM: ...,1,1,1,0  (read=1, write=1, create=1, unlink=0)
TO:   ...,1,0,1,0  (read=1, write=0, create=1, unlink=0)

Verify action_submit still works via sudo() context flag.

ESTIMATED IMPLEMENTATION EFFORT:
├─ Code Change: 1 hour (single line ACL change)
├─ Testing: 1 day (regression testing all workflows)
└─ Total: 1-2 developer days

RECOMMENDED APPROACH: Phase 2 Fix
Include in post-deployment security hardening sprint.

TESTING STRATEGY:
1. Functional Tests: Submission still works
2. Security Tests: Direct write blocked for requesters
3. Negative Tests: Unauthorized edit attempts fail
4. Regression Tests: All requester actions work
5. API Tests: Direct ORM write operations blocked

│ ISSUE CRIT-007: ALLOCATION-TRIP-REQUISITION WORKFLOW FRAGMENTED            │
├─────────────────────────────────────────────────────────────────────────────┤
│ Status: PARTIALLY CONFIRMED - OPERATIONAL INEFFICIENCY                      │
│ Verification Confidence: HIGH (Workflow exists but requires manual steps)  │
└─────────────────────────────────────────────────────────────────────────────┘

ROOT CAUSE:
Workflow connections exist and function correctly, but require too many manual
steps between requisition approval and trip execution. Automation gaps create
operational friction.

EVIDENCE LOCATION:
- Allocation Creation: models/fleet_allocation.py, Lines 193-214
- Trip Creation: models/fleet_allocation.py, Lines 338-375
- State Sync: models/fleet_allocation.py, Lines 220-239
- Manual Trigger: models/fleet_requisition.py, Lines 736-755

TECHNICAL DETAILS:
CONNECTIONS VERIFIED (Working):
✓ Requisition → Allocation link created (allocation_id field)
✓ Allocation → Trip creation (via action_dispatch_vehicle)
✓ Requisition → Trip direct link created
✓ State synchronization via write() override

MANUAL STEPS REQUIRED (Friction Points):
✗ User must click "Create Allocation" button
✗ User must manually fill allocation form
✗ User must click "Assign Vehicle" button
✗ User must click "Dispatch Vehicle" button

Result: 4 manual clicks + 1 form entry = High friction workflow

SECURITY RISK LEVEL: ★☆☆☆☆ LOW (1/5)
├─ No Security Impact: Data integrity maintained
├─ No Authorization Bypass: All checks enforced
└─ Audit Trail: Complete and accurate

BUSINESS IMPACT: ★★★☆☆ MEDIUM (3/5)
├─ User Experience: Poor (too many clicks)
├─ Training Requirements: Higher than necessary
├─ Error Probability: Medium (users may forget steps)
├─ Process Velocity: Slower than optimal
└─ Operational Efficiency: Suboptimal

PRODUCTION READINESS IMPACT: ★★☆☆☆ LOW (2/5)
├─ Does NOT block deployment
├─ Workflow functions correctly (just inefficient)
├─ Can be improved post-deployment
└─ Users can work around with training

RECOMMENDED SOLUTION:
Automate allocation creation and dispatch after requisition approval:

Modify models/fleet_requisition.py _on_approval_approved():
```python
def _on_approval_approved(self):
    for rec in self:
        if not rec.allocation_id and rec.vehicle_id:
            # Auto-create allocation
            allocation = self.env['hagbes.fleet.allocation'].create({
                'request_id': rec.id,
                'vehicle_id': rec.vehicle_id.id,
                'driver_id': rec.traveller.employee_id.id,
                'allocation_date': rec.date_from,
                'return_date': rec.date_to,
            })
            # Auto-assign and dispatch
            allocation.action_assign_vehicle()
            allocation.action_dispatch_vehicle()
```

This reduces 4 manual steps to 0 automatic steps.

ESTIMATED IMPLEMENTATION EFFORT:
├─ Code Changes: 2-3 days (modify approval callback)
├─ Testing: 2-3 days (comprehensive workflow testing)
├─ User Acceptance: 1 day (validate simplified flow)
└─ Total: 5-7 developer days

RECOMMENDED APPROACH: Phase 2 Enhancement
Deploy current version, gather user feedback, implement automation in sprint 2.

ALTERNATIVE APPROACH:
Create a guided wizard that combines all steps in one interface:
- Single "Dispatch Requisition" wizard
- Collects all required information
- Executes all steps automatically
- Provides clear progress feedback

TESTING STRATEGY:
1. Functional Tests: Auto-allocation creates correct records
2. Integration Tests: All workflow steps execute in sequence
3. Negative Tests: Missing data handled gracefully
4. Performance Tests: Automation doesn't create bottlenecks
5. User Acceptance Tests: New flow is faster and clearer
6. Regression Tests: Manual flow still works (backward compatibility)

RISK MITIGATION:

- Provide manual override for edge cases
- Log all automatic transitions for audit
- Add validation checks before each automatic step
